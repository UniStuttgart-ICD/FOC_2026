Make the blackboard remember intent, not just execution.

The current job board can track many queued MoveIt calls. The missing layer is a task ledger above it, so background recovery, new user commands, retries, and cancellations all attach to the right user intent.

Jobs are tool executions. One MoveIt call, one result, one terminal event.
Tasks are user intent. One command may produce several jobs and recovery turns.
The coordinator owns concurrency. The LLM acts only inside a selected task context.

Coordinator Decisions

The same failed background job is handled differently depending on what the user says next. That routing should be deterministic before the LLM is called.

Independent command

Task A fails in the background. User asks, "what is your pose?"

task-a.status = recovering task-b.kind = observation actuator_conflict = false
route without blocking
Keep failed task context intact.
Run the independent turn now.
Resume recovery later if still relevant.

Result

The user gets the pose answer. The old failure stays attached to task-a and does not hijack the visible conversation.

task-b.status = completed task-a.recovery_deferred = true speak(task-b.answer)

Code Shape

Keep the current job board deterministic. Add task ownership above it and route failed jobs through a coordinator instead of directly into a recovery graph turn.

RobotTaskBoard

Stores user-level intent, task status, parent/supersedes links, and which jobs belong to each task.

task_id requested_by_turn_id jobs: list[str] status: recovering | blocked | done

RobotJobBoard

Still owns exact MoveIt calls. The worker must not repair, reinterpret, or merge commands.

job_id task_id tool_name arguments

TaskCoordinator

Classifies new turns against active tasks before deciding whether to run, attach, queue, cancel, or recover.

classify_turn(...) route_event(...) select_task_context(...)

Recovery Turn

Runs only when selected by the coordinator and includes explicit task context, not just a plain user-looking prompt.

source = "task_recovery" task_id = "task-a" job_id = "job-91"

Event Grammar

This is the minimum event vocabulary I would want before letting multiple commands and background recovery run at the same time.

task_createdA user turn becomes a tracked intent.
job_queuedA MoveIt tool call is attached to one task.
job_failedThe failure payload stays on the job and task.
recovery_requestedThe coordinator, not the worker, asks for recovery.
task_supersededA newer command replaces older recovery.
task_cancelledStop/forget commands suppress stale follow-up.
