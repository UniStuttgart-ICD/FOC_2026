Original: direct Vizor ROS stack

  • Runs cxy201/noetic-vizor directly.
  • Starts vizor2ros.launch from the compose command.
  • Exposes rosbridge and Vizor TCP ports only.
  • No desktop, RViz, noVNC, MCP servers, or MTC packages are declared.
  • Uses active:=HOLO1,HOLO2.

Current: RViz plus MCP stack

  • Builds local/noetic-vizor-rviz:latest from a local Dockerfile.
  • Starts Vizor through an entrypoint that also launches desktop and helper ROS nodes.
  • Adds browser-visible noVNC at http://127.0.0.1:6080.
  • Adds agent-facing moveit-mcp and advisory vizor-mcp.
  • Installs RViz, MoveIt visualization, Robotiq, and MTC packages.

What Changed and What Is New

This is the compact checklist version of the comparison.

New capabilities
  • Browser-accessible noVNC desktop for RViz.
  • RViz config for UR10 RobotModel, MoveIt planning scene, and planned paths.
  • MTC RViz panel and installed MTC ROS packages.
  • MoveIt MCP HTTP service at 127.0.0.1:8765/mcp.
  • Vizor MCP HTTP service at 127.0.0.1:8001/mcp.
  • Read-only current pose service for MCP grounding.
  • Robotiq 2F-85 packages integrated into the catkin workspace.
  • Patch step for inherited Vizor robot code and generated UR10-with-gripper URDF.
Behavioral changes
  • vizor-demo is now built locally instead of using only the upstream image.
  • Startup moved from compose command to image entrypoint.
  • Default active Vizor device changed from HOLO1,HOLO2 to HOLO1.
  • MCP servers restart unless stopped; original services did not declare restart policy.
  • Agent integration now uses MCP services rather than direct ROS-only access from outside Docker.
  • MTC proof support is installed but opt-in; real MTC pick construction remains deferred.
Question Answer
Is the current stack still based on the original Vizor image? Yes. local/noetic-vizor-rviz:latest extends cxy201/noetic-vizor; ros-core still runs cxy201/noetic-vizor directly.
Is RViz new? Yes. The original compose did not declare a desktop or RViz. The current image starts RViz in a virtual display and exposes it through noVNC.
Is MTC fully active? No. MTC packages and the RViz panel are present. The proof service is disabled by default, and a real MTC backend remains deferred.
What is the largest architectural addition? The MCP layer: moveit-mcp for robot planning/execution tools and vizor-mcp for advisory Vizor user-sensing context.
What should be watched? The active compose file still lives in the original 01_Docker folder but points to image sources in Multi-Actor-Interface-Library. Keep those paths synchronized when moving machines or repositories.
