MoveIt Tweakable Parameters

This page shows what can be tuned in the current UR10 MoveIt stack, what the agent can pass today, and which settings need a code or container change.

Observed from running Docker containers on 2026-05-18: ros-core, vizor-demo, vizor-mcp, and moveit-mcp.

Short Answer

The stack has many MoveIt knobs, but only a small set is reachable through the current agent tools.

Agent can tune now

timeout_s, poses, waypoints, pick/place distances, grasp face, orientation mode, target position, and planning_strategy.

MoveIt can tune, but agent cannot

Planner pipeline, planner ID, OMPL planner params, execution monitor tolerances, kinematics tolerances, joint limits, and Cartesian interpolation internals.

Hard-coded today

Free motion uses Pilz PTP. Two-pose Cartesian uses Pilz LIN. Sampled approach uses OMPL RRTConnect. Multi-waypoint Cartesian uses eef_step = 0.05 and avoid_collisions = true.

Important distinction

timeout_s is the wait budget in the MCP/Vizor wrapper. It is not the same as MoveIt's internal allowed_planning_time. The current tools do not expose allowed_planning_time or num_planning_attempts as call arguments.

Agent Tool Knobs

These are the parameters the voice agent can pass through the current MCP tools without editing the Docker image.

Setting Current value or range Plain meaning Risk when changed
robot_name UR10 only The only robot namespace allowed by Robot Call Validation. Other names are refused before MoveIt sees the request.
timeout_s Must be greater than 0 and at most 60. Defaults vary: observe 2, gripper 5, planning 10, task execution 60. How long the wrapper waits for a result before saying the call failed. Higher values reduce false timeouts. They also make failed calls take longer.
target_pose {x, y, z} or {position, orientation} The TCP goal pose for free-space planning. Bad pose values make planning fail. Coordinates are limited to +/-1.5 m.
waypoints One or more target poses The ordered TCP path for Cartesian motion. Too-close waypoints can collide. Missing orientation can allow unexpected tool orientation.
plan_name Optional string A label for the plan. The system normally creates one. Reusing names can be confusing; validation usually expects fresh returned names.
grasp_face Default top; must match a face from moveit_get_object_context Which side of the object the gripper approaches. A face with less clearance can fail even when another face is easy.
approach_distance_m Default 0.08; must be positive How far away the approach pose is from the selected object face. Too small starts near collision. Too large may make the connect motion harder.
grasp_standoff_m Default 0.01; must be positive How close the TCP gets to the grasp face before closing the gripper. Too small may be too close to the object collision geometry. Too large may not actually grasp.
lift_distance_m Default 0.1; manipulation validation allows 0.0 to 0.20 How high the object is lifted after grasp. Higher lift can hit limits or nearby geometry. Zero lift is useful for hold-in-place tests.
planning_strategy auto, cartesian, or sampled_approach Which pick planning route to try. sampled_approach uses OMPL/RRTConnect. It can solve some cases the Cartesian route misses, but it is not the default exposed free-motion planner selector.
target_position {x, y, z} The object center for place, move-and-release, or pick-place tasks. Bad target values create impossible or unsafe place poses.
orientation_mode keep, horizontal, vertical, or explicit How the place TCP orientation is chosen. explicit needs a valid quaternion. The wrong orientation can make IK fail.
place_standoff_m Default 0.01; must be positive How far the TCP stays above the object release point. Too small can collide with the target surface. Too large can drop or release in the wrong place.
retreat_distance_m Default 0.1; must be positive How far the gripper backs away after release. Long retreats can hit limits or geometry.
backend staged_moveit for moveit_plan_manipulation_task; mtc for moveit_plan_compound_task Chooses the task planner family. The current manipulation task requires staged_moveit. The MTC compound path fails closed if MTC is unavailable or incomplete.

Current Vizor Planning Routes

These choices are inside the patched Vizor robot code. They are real tuning points, but they are not normal agent-call parameters today.

Route Current planner choice Important constants What to tune
/UR10/request/free Pilz-style PTP planner_id = PTP Planner pipeline, planner ID, planning time, attempts, and velocity scaling would need code exposure.
/UR10/request/cartesian with two poses Pilz-style LIN planner_id = LIN, avoid_collisions = true Useful for simple straight moves. It still depends on IK, collision geometry, and path limits.
/UR10/request/cartesian with more than two poses MoveIt Cartesian interpolator eef_step = 0.05, avoid_collisions = true, jump_threshold = None, success requires fraction >= 1.0 eef_step is the main hidden tuning point. Smaller values check the path more densely.
/UR10/request/sampled OMPL RRTConnect planning_pipeline = ompl, planner_id = RRTConnect This is the current sampled alternative to straight Cartesian approach motion.

Should The Agent Choose These Routes?

Yes, but not as raw ROS topics. The better interface is a small set of route profiles. The agent should say what kind of retry it wants, and Robot Control should map that to the right Vizor route.

Main finding

The agent already reaches the Vizor free and cartesian routes through MCP. The useful route that is mostly hidden from the main manipulation path is sampled, which uses OMPL RRTConnect.

Route profile How it is set today Is the agent already using it? Should we expose it?
free_pilz_ptp moveit_plan_free_motion publishes /UR10/request/free. Vizor sets planner_id = PTP. Yes. Staged manipulation uses it for connect_to_pre_grasp and connect_to_place. Already exposed. It is good for a clean point-to-point move, but it was also the route that failed in the second trace on connect_to_pre_grasp.
cartesian_pilz_lin moveit_plan_cartesian_motion publishes /UR10/request/cartesian. If there are exactly two poses, Vizor sets planner_id = LIN. Yes. Staged manipulation uses it for final approach, lift, place approach, and retreat when those are two-pose segments. Already exposed. Keep it for near-object straight-line moves because it is simple and predictable.
cartesian_interpolator The same /UR10/request/cartesian route switches to compute_cartesian_path when there are more than two poses. It uses eef_step = 0.05 and avoid_collisions = true. Sometimes. It depends on the number of waypoints. The common staged pick segments use two poses, so they use Pilz LIN, not this branch. Expose only through a named Cartesian profile. Do not let the agent freely change collision checking.
sampled_ompl_rrtconnect plan_sampled_motion publishes /UR10/request/sampled. Vizor sets planning_pipeline = ompl and planner_id = RRTConnect. Only in a narrow path: moveit_plan_pick(planning_strategy="sampled_approach"). The main moveit_plan_manipulation_task path does not use it for failed stages. Yes. This is the main route worth exposing as a robust fallback for connect/preposition stages.

Recommended exposure

Agent-facing option Internal behavior Why this is safer
planner_profile = "default" Use the current staged route: free stages through Pilz PTP, near-object stages through Pilz LIN. Keeps current behavior for normal motions.
planner_profile = "robust_connect" Use OMPL RRTConnect for connect/preposition stages, then keep final approach/lift/retreat as straight Cartesian motion. This targets the likely weak spot without changing the whole task. It helps when the robot needs a non-straight or sampled path to reach the approach pose.
planner_profile = "cartesian_only" Use Cartesian/Pilz LIN for simple straight segments only. Useful when the user clearly wants a straight tool motion and the start pose is already good.
planner_profile = "diagnostic_retry" Retry with extra planner evidence: route name, planner ID, MoveIt error code, Cartesian fraction, trajectory point count, and known scene objects. This lets the next retry be based on evidence instead of guessing.

The route profiles should live in Robot Control or the MoveIt MCP layer, not in the model prompt as raw topic names. The agent should not publish /UR10/request/free, /UR10/request/cartesian, or /UR10/request/sampled directly.

Missing From The Current Agent Interface

These settings are real MoveIt concepts, but the current MCP tool schemas do not let the agent pass them directly.

Setting Current access Why it matters
allowed_planning_time Not exposed as an MCP argument. This is MoveIt's internal planning budget. It is different from timeout_s.
num_planning_attempts Not exposed as an MCP argument. A Vizor param named planning_attempts=20 exists. More attempts can improve stochastic planner reliability, especially with OMPL.
planning_pipeline Available in MoveIt. Hard-coded by the Vizor routes. Lets a caller choose OMPL, Pilz, or CHOMP per request.
planner_id Available in MoveIt. Hard-coded by the Vizor routes. Lets a caller choose RRTConnect, PTP, LIN, and others per request.
eef_step Hard-coded to 0.05 in the multi-waypoint Cartesian path. Controls how densely MoveIt samples a straight Cartesian path.
jump_threshold Not passed in the current patched Noetic path; diagnostics log None. Controls rejection of sudden joint jumps along a Cartesian path.
avoid_collisions Hard-coded to true for Cartesian interpolation. Turns collision-aware Cartesian interpolation on or off. Turning it off should not be a normal agent option.
Velocity and acceleration scaling per request Configured globally in MoveIt params. Not exposed as MCP tool arguments. Useful when slow robot execution is expected or when paths need gentler motion.

Knobs Worth Exposing To The Agent

These are the main candidates to expose if we want the agent to tune simple motions instead of only retrying the same request. The safest shape is usually a small set of named profiles, not raw free-form numbers.

Recommended rule

Let the agent choose a profile such as fast, robust, precise, or diagnostic. Keep hard safety choices, like collision checking, controlled by the system.

Candidate knob Suggested agent shape Why it could help simple motions Risk
planning_pipeline and planner_id planner_profile: pilz_ptp, pilz_lin, ompl_rrtconnect, cartesian A simple pose can fail in one planner and succeed in another. For example, a strict straight-line Cartesian move may fail near an object, while OMPL can route around it. Raw planner names give the agent too many choices. Use a short allow-list.
allowed_planning_time planning_budget: short, normal, long Some OMPL plans need more search time. A long budget can turn a false failure into a valid path. Failed plans take longer to return. This should be used only after a real planning failure or for complex scenes.
num_planning_attempts attempt_budget: 1, 3, 5 OMPL is sampled. More attempts can find a path when one random search misses an easy solution. More attempts can add wait time and CPU cost. It should pair with a clear total timeout.
goal_position_tolerance and goal_orientation_tolerance goal_precision: exact, normal, loose The current kinematics tolerances are tight. A simple target can fail if the requested pose is slightly over-constrained, especially orientation. Loose tolerance means the final TCP pose may not be exactly where requested. Use only when task semantics allow it.
kinematics_solver_timeout ik_budget: current, more The Docker patch now sets 0.05 seconds. The previous observed value was 0.005 seconds, a tiny search window. More IK time can help near joint limits or awkward orientations. More IK time can slow each waypoint check, especially Cartesian paths with many samples.
eef_step cartesian_resolution: coarse, normal, fine Controls how densely a straight Cartesian path is checked. Fine checking catches real collision or IK problems. Coarse checking can be faster for simple open-space motion. Too coarse can hide a collision between samples. Do not use coarse near objects.
jump_threshold joint_jump_check: off, normal A straight TCP path can make the joints flip. A jump check can reject that. Turning it off can help diagnose whether jump checking is the reason for failure. Disabling jump checks can allow ugly or unsafe joint motion. It should be a diagnostic or tightly controlled option.
trajectory_execution/allowed_execution_duration_scaling and allowed_goal_duration_margin execution_patience: normal, slow_robot This directly matches the timeout symptom: the robot may still be moving, but MoveIt or the wrapper decides it took too long. Too much patience can delay detection of a real stuck execution.
trajectory_execution/allowed_start_tolerance start_tolerance: strict, normal, loose If the real robot or fake controller starts slightly away from the expected first point, execution can be rejected before useful motion begins. Too loose can start a trajectory from a state that is not close enough to the planned path.
Per-request velocity and acceleration scaling motion_speed: slow, normal, fast_sim Slower motion can be easier for the real robot to track. Faster simulation can reduce false waiting in RViz-only tests. Changing speed changes execution time. The timeout should scale with the selected speed.
approach_distance_m, grasp_standoff_m, and place_standoff_m profiles clearance_profile: near, normal, extra_clearance Many simple object motions fail because the TCP or gripper is too close to collision geometry. A little more standoff can make the plan feasible. Too much standoff may miss the object or release it in the wrong place.
Start-state repair knobs: jiggle_fraction, start_state_max_bounds_error start_state_repair: off, normal, more If the current state is barely invalid, a small nudge can let planning start instead of failing immediately. This should stay small. Large nudges can make the plan start from a state the robot is not really in.
Failure-detail logging diagnostic_level: normal, planner_debug The agent needs to know whether a failure was IK, collision, timeout, start-state mismatch, or incomplete Cartesian fraction. Better evidence makes the next knob choice targeted. More logging can be noisy. It should be enabled around failed plans, not all the time.

Most useful first profiles

Profile What it would change When the agent should use it
slow_robot_execution Raise execution timeout, raise execution duration margin, scale timeout from trajectory duration. Use when the plan exists and preview/execution starts, but the final result times out.
ompl_robust_connect Use OMPL RRTConnect, longer planning time, and several attempts. Use when Pilz PTP or a direct Cartesian route fails for a simple reachable target.
cartesian_precise_near_object Use fine eef_step, keep collision checking on, keep strict Cartesian fraction checking. Use for final approach or retreat near an object, where skipping a collision would be bad.
relaxed_pose_goal Use slightly looser position and orientation tolerances, especially orientation. Use when the exact quaternion is not important and IK appears over-constrained.
extra_clearance_pick Increase approach distance and standoff by a small fixed amount. Use when failures happen close to an object or table and the grasp does not require being that close yet.
planner_debug_once Log Cartesian fraction, MoveIt error code, collision contacts if available, IK failure hints, planner pipeline, planner ID, and trajectory point count. Use after one failed simple motion before changing many knobs blindly.

Likely Best First Tuning Points

If the goal is better reliability without a rewrite, these are the practical places to start.

Priority Knob Reason
1 timeout_s for execution and long planning calls This directly addresses false failures when the robot is simply slow.
2 approach_distance_m and grasp_standoff_m These change how close the TCP gets to object collision geometry.
3 Expose planner profile for planning_pipeline, planner_id, planning time, and attempts This gives controlled access to OMPL/RRTConnect without rewriting the whole system.
4 Expose eef_step only as a small set of profiles It is useful, but a free numeric knob can hide bad geometry if made too coarse.
Avoid Turning off avoid_collisions That can make a path plan by ignoring the reason it was unsafe.
