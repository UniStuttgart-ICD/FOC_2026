Read This First

  • Voice Runtime does not know LangGraph, MCP, MoveIt, or robot policy internals.
  • Agent Control satisfies the Agent Turn backend seam and uses Robot Control.
  • Robot Control exposes structured robot tool feedback to Agent Control.
  • MoveIt planning and execution gates robot motion.
  • Verified real robot execution sends URScript and Robotiq commands outside MCP.

Agent Control drilldown

LLM Input Stack

One model call is not just the transcript. Agent Control builds a typed turn, refreshes advisory context, records completed worker results into Robot Context, renders the Job Board prompt summary, filters task-solution execution tools for the active mode, then sends a system message, user message, current context blocks, and tool schemas into the provider-backed chat model.

Solid path = normal model request and response. Dashed path = robot result, correction, or retry loop.

Async job lifecycle

Robot Job Board Lifecycle

When the model calls a queueable robot action, Agent Control submits the exact tool call to the Job Board. The Job Worker claims queued work, executes it through MoveIt MCP or verified execution, records a terminal event, and the next turn sees a compact prompt summary block.

Solid path = job submission, queueing, and execution. Dashed path = terminal state reused for notification and next-turn context.
Robot Context

Records completed job results after terminal events; it is advisory state, not the queue.

Task Policy V1

  • Fresh pose before movement, planning, retries, or relative commands.
  • No blind moveit_execute_plan; use a recent executable plan.
  • Basic gripper and attach ordering before attaching objects.
task-solution contracts

MTC-backed planning and compound execution

Dynamic pick stays on the semantic task path. Compound manipulation adds the same discipline: the model requests moveit_plan_compound_task with hard requirements and optional preferences; stage_intents are hints only. MCP/MTC compiles and solves the task. Pipecat executes only a recent approved task_solution_id with a supported proof-backed execution_contract.

configured success path fail-closed or retry path Stage retry stays inside moveit_execute_task_plan; it is not fallback to another backend.
Default is not fallback

backend="emulated" remains the default configuration. If MTC is configured and fails, the response is structured failure, not emulated planning.

Why dynamic_1 changes

Dynamic or vertical objects now expose multiple candidate attempts across grasp faces and distances, so one low side/back approach no longer represents the whole search.

When the agent may say success

Success requires verified execution, physical gripper close, MoveIt attachment sync, and attachment verification evidence. Otherwise Robot Context does not mark the object held.

supported contract path fail-closed correction path The LLM states requirements/preferences; MCP/MTC owns compilation and solving.
Current service boundary

/vizor_mtc/plan_compound_task is wired as the ROS/MTC compound boundary. It must fail closed until real typed compound task construction returns solved stage evidence.

Contract fields

Each step needs a handler, source stage, required proof, object context, scene snapshot, and motion or release evidence where that handler needs it.

Unsupported actions

Contact manipulation such as slide or push is rejected at plan time. The executor only runs supported task kinds and typed handlers.

Physical actuation boundary

Verified Real Robot Execution

The real UR10 path starts after MoveIt has produced an executable plan. The verified execution server is a host-side actuation adapter: it executes cached MoveIt trajectories, homes the arm, and controls the Robotiq gripper without becoming another MCP planning surface.

MoveIt plan cache -> verified execution server -> URScript 30002 + Robotiq 63352

Owns

Cached plan execution, home, direct Robotiq gripper commands, startup readiness, HTTP command responses, and operator-dashboard actuation hooks.

Does Not Own

MoveIt planning, semantic task policy, agent tool choice, RViz visualization, or broad ROS topic mutation.

Channel Rules

  • Plan through MoveIt MCP before physical execution.
  • Use moveit_execute_task_plan for verified supported task solutions.
  • Keep moveit_execute_task_solution for sim/emulated task execution only.
  • Execute only a recent plan name or explicit operator command.
  • Send arm motion/home as URScript over 30002.
  • Send gripper open/close over Robotiq 63352.
  • Use RTDE Receive for readiness only; avoid RTDE Control in production actuation.

Host Components

  • server/verified_execution_server/
  • server/operator_dashboard/
  • server/moveit_mcp/

Why This Boundary Exists

  • MoveIt remains the planning and execution gate the agent understands.
  • The physical server is intentionally simpler than the old robot MCP: no extra planning tools, no broad robot API, no agent-facing improvisation.
  • URScript avoids RTDE Control register ownership conflicts while still letting the real robot run the planned joint program.
  • Robotiq gripper control stays independent from arm motion and can be tested at server startup without moving the arm.

What The Agent Should Remember

  • Movement is routed through MoveIt workflows; broad ROS control stays out of Agent Orchestration by default.
  • Physical arm motion uses URScript on 30002; gripper commands use Robotiq 63352.
  • User sensing helps resolve references like "this", "that", "there", and "near me".
  • Missing, stale, or low-confidence user sensing is not proof of user intent.
  • RViz visualizes the planning scene; it is not where agent decisions are made.
