Shared Geometry Model

Chosen direction

Use common_model.json as the canonical beam inventory and editable graph state. Rhino/Grasshopper, ROS, RViz, and AR should be projections of that same body-and-pose model, not separate sources of truth. Current model: six_element_b_c_triangular_frame.

Core thesis

Do not ask the LLM to be a CAD kernel. Give it a compact, readable structure so it can choose the right operation, then let deterministic tools do the exact math.

System responsibility

The shared model is the source of truth. Every other system reads from it, writes validated updates back to it, or projects it into its own runtime format.

Solid lines are state/projection flow. Dashed lines are solved updates and feedback. Grasshopper reads the same body/pose truth through an adapter.

Physical ID vs semantic role

Give every beam a stable physical ID at startup, like element_01. Family names such as B or roles such as shared_vertex_support stay editable metadata, not the primary body identity.

End and direction names

Use the axis as the canonical language: end_start and end_end are stable, while end_neg_x and end_pos_x remain local-frame aliases. Human aliases such as top, left, or shared vertex can be derived from pose.

Grasshopper advice

Use the same common_model.json for beam dimensions and center poses. Let Grasshopper generate Breps from solid.dimensions plus pose; keep Grasshopper layer names and baked object IDs in a derived adapter output.

Fabricability check: shared centerlines create a solid clash

The current graph correctly says element_04.end_start and element_05.end_start share a front vertex, but the top view shows the physical problem: two 4 cm wide rectangular beams cannot occupy the same wedge of space near that vertex. Treat the shared point as a logical joint, then solve separate physical endpoints.

model gap
Recommendation: logical joint + physical setbacks

Keep the abstract shared vertex P, but do not use it as the solid endpoint. Store separate physical starts for element_04 and element_05, offset along their axes until the 4 cm rectangular volumes clear each other.

Option A: move the logical apex beyond the B top

This matches the rotation idea: the beams aim at a virtual point beyond the support, while their physical end caps sit back on valid bearing patches over the B top. The solver must preserve minimum bearing length on element_01.face_top.

Option B: symmetric endpoint setback

For two C beams meeting at angle theta, solve d >= (0.04 + clearance) / (2 sin(theta/2)). The result is a deliberate V gap: same logical joint, separate uncut end caps.

Option C: asymmetric endpoint setback

Let one beam sit closer to P and move the other farther back when one member needs a better bearing patch. This is fabricable, but less visually symmetric.

Rejected: face-axis centerline

Making the centerline run along an inside face hides the clash but pollutes the meaning of beam axis. Keep centroid axes for solids; use guide curves only for logical joints.

No-solution case

With no endpoint offset, no connector, no vertical layering, and one shared coplanar physical endpoint, two untrimmed 4 cm rectangular beams cannot both fit. The solver should reject it.

Best no-extra-part solution

Virtual apex + endpoint setbacks

The joint point becomes digital only. The two C beams still aim at the same virtual apex, but their real rectangular solids stop before that point. This preserves the intended V geometry without adding a cap, trimming the beams, or pretending that two 4 cm solids can occupy the same space.

Why this fits the goal

No new physical part is introduced. The fix lives in the digital model: a virtual joint plus computed endpoint positions.

What the agent sees

joint_front is still one graph node. Each beam stores a relationship to it plus a real endpoint setback.

What the solver does

It finds d1 and d2 so rectangular solids clear each other while preserving the target beam axes.

What validation proves

No beam-beam collision, enough clearance, valid bearing on the B top, and unchanged 4 x 4 cm sections.

Useful when one beam needs priority

Asymmetric setback

Symmetric setbacks are clean, but not always best. If one C beam needs more reliable bearing, better robot access, or a stricter visual alignment, keep that priority beam closer to the virtual apex and push the other beam farther back until the rectangular volumes clear.

When to use it

Use it when one member carries more load, needs more visible alignment, or must remain easier for the robot to grasp.

What changes

The joint graph stays the same. Only d1 and d2 differ, with a reason stored in the model.

What the agent can say

element_04 is the priority member; element_05 yields by moving farther from the apex.

What the validator checks

The priority is respected, the secondary beam clears it, and both beams still have acceptable support conditions.

06

Where to implement it

Put the model where robot-side spatial context belongs: under Robot Control. Keep Rhino/Grasshopper and ROS as projection adapters beside the existing MoveIt MCP.

The core model stays pure and testable. Grasshopper builds display geometry from center pose and dimensions; MoveIt remains the movement-safety boundary.
07

Risks and mitigations

Risk Sev Mitigation
The LLM treats the graph as physical truth. High State clearly that the graph is assembly intent. Collision, reachability, support, and stability must be checked by deterministic tools.
common_model.json, Grasshopper, and ROS state drift apart. High Make common_model.json canonical. Treat Grasshopper Breps, RViz markers, and collision objects as regenerated projections.
The model becomes a raw render asset dump. Med Keep primitives, poses, and generated features canonical. Store Rhino object IDs, GLB, USDZ, MarkerArray, and CollisionObject as derived adapter outputs.
The first schema tries to support every geometry type. Med Start with boxes/beams, generated features, basic contacts, and subassemblies. Add more solids only when the beam workflow is stable.
Implementation leaks geometry math into agent prompts. Low Expose narrow tools like geometry_apply_operation and geometry_solve_grasp_pose; keep math inside shared_geometry.
08

Next decisions

How much history should the file store?

Recommendation: keep current state plus a short operation history. Full replay can wait until the schema proves itself.

Should Grasshopper use the same file?

Yes for v0.1. Let it read common_model.json and generate Rhino geometry from solid.dimensions plus pose. Add a separate generated projection file only when Grasshopper needs bake IDs, layers, or mesh caches.

What does bearing_contact mean?

It means a named supporting feature bears on another named feature within tolerance. In the current file, element_06.face_bottom bears on element_04.face_top and element_05.face_top.

Which operations are v0.1?

Start with set_alignment, move_body, propose_contact, satisfy_constraint, and solve_grasp_pose.

Where does ambiguity resolution live?

The LLM may resolve "those two beams" using user sensing or selection context, but the geometry tool should reject unresolved references.
