LangGraph deep dive

Agent Orchestration runs the robot step loop.

In this codebase, Agent Orchestration is the LangGraph StateGraph behind one Voice Runtime Agent Turn. It observes state, calls the model, executes one robot tool at a time, applies Robot Control policy and validation, and stops only when it has a final answer, a queued job, a verified task result, or a structured blocker.

StateGraph shape

The loop is a small state machine with explicit exits.

The graph is compiled in LangGraphRobotAgent._compile_graph(). Every turn starts with observe_current_pose, then either shortcuts into pending-plan execution or calls the model. Tool results loop back through observation until the graph reaches a final response, a queued robot job, or a hard stop.

START -> observe_current_pose -> call_model -> execute_robot_tool -> observe_current_pose ... -> END
Normal route Correction, repair, or loopback Observation is repeated after robot tool results so the next model call sees refreshed state.
StateGraph Primitive

StateGraph(RobotAgentState) owns the routes. InMemorySaver checkpoints the turn state.

One Tool Rule

Only the first tool call from a model message executes. Extra tool calls get structured retry feedback.

Loop Cap

MAX_AGENT_TOOL_TURNS is 6. Past that, tool use stops and the response becomes a blocker.

Trace Shape

Every graph turn, node, model call, task policy decision, validation result, MCP call, and context update can be traced.

Policy and validation gates

Robot tools pass through deterministic gates before motion.

The model proposes tool calls, but local gates decide whether the concrete step is even eligible to run. These gates are not the movement-safety boundary. They are ergonomic preconditions, shape checks, freshness checks, and proof checks before MoveIt or verified execution is called.

model tool call -> Task Policy -> Robot Call Validation -> Agent Control executor or Robot MCP Bridge
Allowed path Structured blocker path MoveIt and the robot simulation stack remain the motion safety boundary.
Task Policy
  • Requires fresh pose for ordinary motion and moveit_execute_plan.
  • Blocks blind execution of unknown or stale plan_name values.
  • Requires explicit user execution intent before moveit_execute_plan.
  • Redirects compound manipulation to moveit_plan_manipulation_task.
  • Blocks standalone contract-internal scene tools.
Robot Call Validation
  • Allows only canonical MoveIt tools and mapped legacy names.
  • Only robot_name="UR10" is accepted.
  • Target coordinates must be finite and within +/-1.5 m.
  • timeout_s must be finite, positive, and no more than 60 s.
  • Task goals, target requirements, hints, and contract fields are schema checked.
Task Solution Proof
  • Execution needs a recent matching task_solution_id.
  • Explicit approval expires after 60 s or newer user intent.
  • Scene snapshot mismatch blocks execution.
  • Contract steps need supported handlers, source stage, and required proof.
  • Unknown handlers fail closed instead of being interpreted by the model.

Task and job management

There are three kinds of task memory, and they do different jobs.

Robot Context is advisory state for the next model call. Task Solution approval binds a solved manipulation plan to explicit execution intent. The Robot Job Board is the async work queue for longer robot actions. The worker remains deterministic: it executes the exact submitted tool and arguments.

Robot Context is not the queue. The Job Board is not proof. Task Solution approval is not a planner.
01 observe moveit_get_current_pose updates robot name, TCP pose, gripper, and last execution facts.
02 plan Executable plans enter Robot Context for 120 s. Task solutions enter recent-task state with scene snapshot evidence.
03 approve Explicit user intent records approval for the matching task_solution_id and current intent revision.
04 queue Queueable actions become RobotJob entries with exact tool name, args, turn id, and optional continuation.
05 work The worker claims the next job, runs MCP or verified execution, and records completed, failed, or cancelled.
06 notify Completed jobs update Robot Context. Failed planning jobs can become recovery turns; failed execution reports a blocker.
07 summarize The next prompt includes compact Robot Context and Robot Job Blackboard summaries.
Robot Context Store

Tracks advisory current pose, gripper, held object, pending executable plans, recent task solution, recent task failure, and pending task approval.

Robot Job Blackboard

Tracks queued, running, completed, failed, and cancelled jobs plus ordered terminal events.

Prompt Summary

Renders recent job facts into the system prompt. Completed job results are marked as recorded or not yet recorded in Robot Context.

fresh pose 15 s default freshness window before ordinary motion.
plan name 120 s default pending executable plan window.
gripper 30 s default known-closed window before attach.
task approval 60 s approval window bound to task solution and intent revision.
user sensing 2 s default max age when the Vizor sensing bridge is active.

Unified task execution

moveit_execute_task is executed by Agent Control, not by raw MCP.

The single model-visible task execution call is a clean surface. Internally, Agent Control reads the cached task solution, normalizes the execution contract, checks approval, then walks typed stage handlers. Motion stages plan concrete MoveIt plans before execution. Gripper, attach, release, and verification stages require proof-bearing order.

task_solution_id -> approval check -> execution_contract steps -> motion/gripper/scene/proof -> result facts
Supported execution-contract path Fail-closed stage blocker Release/place updates the physical model only when verified release proof contains full object pose evidence.
Supported Task Kinds

pick, place, hold, move_and_release, and pick_place. Unsupported kinds such as slide-style workflows fail before stage execution.

Supported Handlers

motion, close_gripper, open_gripper, attach_object, release_object, verify_attached_object, and verify_released_object.

Motion Stage Behavior

Each motion stage observes current pose, plans a concrete free or Cartesian MoveIt plan, executes it, consumes the pending plan, and retries stage planning/execution up to two attempts.

Failure Behavior

A stage failure records recent task failure evidence, optionally calls moveit_explain_motion_failure, and returns a structured blocker rather than inventing a new task.

Blockers and recovery vocabulary

Most failures become structured correction data, not silent fallbacks.

The loop is intentionally explicit about blocked states. A blocker can be returned to the model as a ToolMessage, shown to the user as final text, stored as recent task failure evidence, or recorded as a job terminal event.

ok=false + error + correction + retryable + suggested_next_tool
Blocker Where It Comes From What The Loop Does
fresh_pose_required Task Policy blocks ordinary motion or execution without a recent robot observation. Suggests moveit_get_current_pose and lets the graph loop through observation.
unknown_or_stale_plan moveit_execute_plan lacks a recent pending executable plan_name. Requires planning first; no invented plan names are accepted.
execution_requires_intent User did not explicitly ask to execute, proceed, confirm, or similar. Blocks execution and asks for explicit approval instead of treating planning as execution.
contract_internal_tool Model tries to call release, release verification, or scene removal directly. Redirects to moveit_execute_task; contract-internal steps run only inside cached task execution.
approval_missing_or_stale Task solution approval is missing, expired, for another solution, changed by newer intent, or scene-stale. Blocks task execution and asks for fresh explicit approval or replanning.
unsupported_contract Execution contract has missing source_stage, missing required_proof, unsupported proof, or unsupported handler. Fails closed and asks for a supported backend task solution.
queued_is_not_evidence Robot Job Submitter returns status="queued" and a job_id. Final text says the job was queued or started, not that the robot moved or completed.
physical_unavailable Verified real robot server, robot, or gripper is unavailable for the optional physical branch. AR/RViz execution can still complete; the response reports physical status separately.
