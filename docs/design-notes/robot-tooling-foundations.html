Pick Needs A Staged Start

The pick planner should not begin with a likely-to-fail long Cartesian approach from home. It should first get near the object with free motion, then use Cartesian motion only for the local grasp interaction.

Updated direction

Approach from a beam-aware staging pose.

Object context should decide the first useful pose. Horizontal beams stage above the object so the gripper can descend locally. Vertical beams stage to the side so the gripper can approach the long face without sweeping in from home. This makes the first motion a normal MoveIt free-motion plan, not a brittle Cartesian path across the workspace.

RuleLong travel to the object uses free motion from the start.
RuleOnly the local approach, grasp, and lift segment is Cartesian.
RuleThe tool returns structured guidance instead of hiding a required preposition step.

Stage 1: near-term fix

moveit_plan_pick(auto) should stage first. Instead of trying a long Cartesian pick from home and learning the obvious failure, it derives a beam-aware staging pose from object context, plans that pose with free motion, and returns the staging plan as the next executable action.

Compute a staging TCP pose from the object, not from a generic home-to-grasp line.
For horizontal beams, stage above the beam with the gripper pointing down.
For vertical beams, stage beside a long side face instead of using the top face.
Use free-motion planning to that staging pose, so MoveIt can solve the whole-arm travel.
Return the preposition plan name as executable, with retry guidance for the local Cartesian pick.
Ask the agent to execute the staging plan, re-read pose/object context, then retry moveit_plan_pick for the local approach.

Stage 2: proper workflow fix

The proper fix is a workflow executor below the agent. It still keeps planning and execution gated, but the robot layer owns the pick sequence after explicit execute intent, instead of asking the model to manually chain gripper, attach, verification, and lift steps.

Execute the verified sequence as preposition -> local approach -> close gripper -> attach -> verify -> lift.
Preposition with free motion to the beam-appropriate staging pose.
Run the local Cartesian approach only once the arm is staged near the object.
Close the gripper, apply the attached collision object, then verify through /UR10/get_planning_scene.
Treat attach as complete only when the object is attached to the gripper link in the service-read scene.
Return verified completion or a structured blocker with the failed step and next recovery tool.

Beam orientation decides the staging side

Horizontal beams stage above the object. Vertical beams stage to the side. The planner should choose that pose before asking for Cartesian approach waypoints, because local Cartesian interaction is useful only after the robot is already near the object.

Current live checks show the horizontal top-stage preposition succeeds. Vertical side staging is now correctly selected, but still needs tuning: the present side pose can fail with no IK solution.

ReAct Execution Loop

The current tools are shaped for an agent that observes, chooses a grounded action, receives verification feedback, and then either continues or stops with a concrete blocker.

Observe

Read task context

moveit_get_robot_state, moveit_list_scene_objects, and moveit_get_object_context give the agent current robot and object facts.

Reason

Choose the next tool

The model decides whether it has enough information to pick, replan, ask, or diagnose.

Plan

Ask MoveIt for feasibility

moveit_plan_pick, moveit_plan_place, Cartesian planning, and free-motion planning return executable plans, staging guidance, or structured blockers.

Act

Execute verified workflows

Execution goes through moveit_execute_plan with a recent pending plan and explicit execute intent.

Verify

Continue or stop clearly

Results must include ok, verification checks, correction text, retryability, and a suggested next tool when useful.

What The Current Tools Can Compound

Compound behaviors are still useful. The issue is where the composition happens. Agent sequencing is good. Robot-side primitives need to match the intent class.

works well

End-effector gestures

Wave, trace a line, lift slightly, draw a small shape, move to a target pose.

fragile

Body-like motion via TCP arcs

Base turns, posture shifts, elbow movement, or “turn around” can be approximated but not guaranteed.

still missing

Joint-space intent

The agent needs a typed way to ask MoveIt for planned joint changes, not raw ROS mutation.

moveit_get_robot_state Is Necessary, Not Sufficient

Robot state answers “what is true now?” A failure-explanation tool answers “why did this specific attempt fail, and what should the agent try next?”

moveit_get_robot_state

This is implemented as readiness and verification context, not as the pick capability itself.

Current pose and planning frame.
Physical-mode flag.
Latest fake-controller joint state.
Useful before pick planning and after a pick attempt to compare expected and observed state.

Now exposed: moveit_explain_motion_failure

Use it after a failed plan, failed execution, or failed verification when the agent needs a grounded next move.

Consumes the failed tool result and original arguments.
Classifies failures such as incomplete path, unreachable waypoint, stale state, or verification mismatch.
Returns concise correction text and a suggested next tool.
Turns planner evidence into agent-ready recovery guidance.

Execution Gate

This page must be read together with MoveIt to Real Robot Execution Check. The current runtime is simulation-only, but the tool contract still plans first and executes only through one gated executor.

planner

moveit_plan_pick

Creates a pick plan or returns a required staging plan. Robot Context remembers successful returned plan names as pending executable plans.

confirmation

Explicit execute intent

Execution requires a current user command such as “execute it” or “go ahead.”

executor

moveit_execute_plan

The executor consumes the exact pending plan name, checks readiness, moves, and reports verified completion or failure.

moveit_plan_and_execute_* must not be a tool

The robot tool contract keeps planning and execution as separate verbs. Do not expose combined plan-and-execute tools for pick, place, free motion, or Cartesian motion. A planner returns a plan name; moveit_execute_plan consumes that exact recent plan only after explicit execute intent.

Architecture Rule

The references in ARCHITECTURE.md point in the same direction: introspect first, expose typed capabilities, and verify execution.

Narrow tools

Do not expose broad ROS control or raw topic mutation to the agent.

Robot owns checks

Readiness, planning, execution verification, and hard limits stay below the agent.

Agent owns sequence

The agent chooses the next semantic tool, based on tool feedback and user intent.

Failure is data

Failed plans should become structured recovery signals, not vague apologies.

Anthropic Tool-Writing Guidance

Anthropic’s agent-tool guidance says tools are contracts between deterministic systems and non-deterministic agents. For this project, that means robot tools should be designed around task affordances, not raw APIs.

choose

Few high-impact tools

Keep pick/place workflows ahead of broad low-level scene, topic, or service wrappers.

namespace

Clear boundaries

Group tools by resource and intent: state, scene object, planning, pick, execution, diagnosis.

context

Return what guides action

Use names, poses, frames, grasp candidates, correction text, and suggested next tools over raw IDs alone.

budget

Keep responses concise

Default to compact summaries, with optional detailed output for debugging and replay.

spec

Write for a new teammate

Tool descriptions should name when to use the tool, required inputs, outputs, and recovery behavior.

eval

Test with real tasks

Evaluate prompts like “pick up the red beam” and inspect calls, errors, latency, and verification evidence.

Source: Anthropic Engineering, “Writing effective tools for agents — with agents.”

Remaining gaps: possible tools

These are not current Pipecat-agent tools. Add them only if the behavior belongs below the agent and can return concrete planning or verification evidence.

Joint-space body movement. Candidate: moveit_plan_joint_motion for typed joint targets or bounded deltas.
Base-turn voice intent. Candidate: moveit_rotate_base as a narrow semantic wrapper over base-joint planning.

Reference Takeaways

These are not dependencies. They are design pressure toward agent-friendly robotics boundaries.

ROSA

Introspect first

Give the agent state and diagnosis tools before it improvises.

RobotMCP

Typed ROS boundary

Expose topic, service, and action behavior through MCP-level capability tools.

RAI

Readiness gate

Status and readiness should shape what motion tools are available or advisable.

ROS-LLM

Structured execution

Keep behaviors as explicit steps over ROS actions and services.

APYROBO

Capability plus replay

Semantic capabilities, safety policy, observability, and replay belong together.

Realtime MCP

Tool narrowing

Give the model the smallest useful tool set for the current context.
