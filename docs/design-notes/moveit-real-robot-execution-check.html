Investigation memo - May 14, 2026

MoveIt should plan. One gated executor should move.

The current stack already has a verified MoveIt planning contract and a direct ur_rtde robot MCP. The next step is not to let the agent jump between them freely. The real-robot path should preserve the named-plan gate and route only verified trajectories into the RTDE executor.

Verdict

Reuse the existing ur_rtde UR bridge behind a new verified-plan execution adapter. Do not expose low-level RTDE motion tools as the agent's normal motion path.

What is already in the codebase

MoveIt

Planning is already named and verified

moveit_mcp exposes planning tools, stores verified plans in process memory, and only executes a plan name that came from a successful planning call.

moveit_mcp/server.py: canonical tools include moveit_plan_free_motion, moveit_plan_cartesian_motion, and moveit_execute_plan. moveit_mcp/tools.py: MoveItMcpTools._planned stores executable plan summaries.
ROS

The execute event is a ROS topic

The actual execution trigger is std_msgs/String on /UR10/command/execute. The message data is the selected plan name.

moveit_mcp/vizor_client.py: publishes {"data": plan_name} to /UR10/command/execute. Inside the Vizor container, Robot.execute() looks up trajectory_data[plan_name].
Robot MCP

The RTDE path exists, but it is low-level

mcp_server/ runs a Robot MCP at 127.0.0.1:8766/mcp. It can directly call moveJ, moveL, moveJ_IK, servoJ, and execute_trajectory through ur_rtde.

mcp_server/server.py: Streamable HTTP FastMCP server. devices/ur/bridge.py: RTDE control, receive, IO, gripper, and trajectory execution.
UI

The local operator dashboard is not the motion UI

The MAVE operator dashboard starts and stops services. It does not contain a robot motion execute button. A legacy VizorWebControl button exists, but it targets /KR210/command/execute with "Jog", not the UR10 named-plan path.

operator_dashboard/static/app.js: service lifecycle calls only. VizorWebControl/scripts/controls.js: legacy KR210 jog execute.
Payload contract

Do not confuse a plan name with RTDE frames

Direct answer folded in: running the Robot MCP server is necessary for a direct RTDE path, but it is not enough by itself. MoveIt MCP returns a verified named plan; Robot MCP expects explicit trajectory frames.
Component Current contract Execution payload
MoveIt plan Plans through /UR10/request/free or /UR10/request/cartesian. Returns metadata such as raw.plan_name, trajectory_points, can_execute, and final joint positions. It does not hand the agent a Robot MCP frame sequence.
Vizor execute Receives /UR10/command/execute with the selected plan name and looks up trajectory_data[plan_name]. In physical mode, forwards a trajectory_msgs/JointTrajectory toward /UR10/execute.
Robot MCP Uses the UR bridge and ur_rtde for direct execution. execute_trajectory(frames) expects frames like [{"positions":[j1,j2,j3,j4,j5,j6]}].
Executor Must choose one backend boundary instead of mixing contracts in the agent. Either execute by plan name through Vizor, or fetch/retain the planned trajectory and convert it to Robot MCP frames behind the verified executor.
Two allowed triggers

How execution should be triggered

Option A: AR execute button Human trigger
  • The agent or UI plans with MoveIt and receives raw.plan_name.
  • The AR interface shows a pending plan to the user.
  • The execute button publishes std_msgs/String to /UR10/command/execute.
  • The payload must be the exact plan name, not a generic string such as "Jog".
  • For physical execution, the robot name and /UR10/execute route must reach the RTDE-backed controller.
Option B: Agent execute tool Voice trigger
  • Use the existing moveit_execute_plan tool name.
  • Require a pending verified plan in robot context.
  • Require explicit current user intent: for example, "execute it" or "go ahead".
  • If the tool bypasses Vizor and calls Robot MCP, the executor must fetch or retain the full trajectory and convert it to execute_trajectory(frames).
  • Check physical mode plus emergency-stop and protective-stop status before the RTDE call.
  • Consume or expire the pending plan after execution to prevent stale repeats.
Important: if explicit confirmation is mandatory, the combined moveit_plan_and_execute_free_motion and moveit_plan_and_execute_cartesian_motion tools must be removed from the normal prompt/tool surface or policy-gated too.
Recommended integration

A single verified-plan executor

1

Observe

Agent calls moveit_get_current_pose when motion depends on state.

2

Plan

MoveIt receives /UR10/request/free or /UR10/request/cartesian.

3

Register

The successful plan is remembered as a pending executable plan.

4

Confirm

AR button or explicit user command selects the exact plan_name.

5

Execute

The executor sends the verified trajectory through Vizor or an RTDE frame adapter and reports completion.

Do not do this

Do not let the agent plan with MoveIt and then freely call low-level RTDE tools.

Why

That splits planning and execution authority. It would bypass the named-plan check, recent-plan state, and MoveIt trajectory verification. The RTDE code is reusable, but it should sit behind the verified executor boundary.

Job board stays the ledger

RobotJobBoard should keep queued, running, completed, failed, raw result, job id, turn id, and user text. It is the audit trail for worker activity, not the policy authority for execution permission.

Robot context owns pending plans

RobotContextStore should hold the policy-facing pending executable plan record: plan_name, source tool, source job, timestamp, user request, and consumed or expired state.

Completion bridge materializes state

On successful worker completion for moveit_plan_*, extract raw.plan_name and materialize it into robot context. Execution checks then read context, while diagnostics still read the job board.

Simplified server plan

Build one narrow executor, not another robot toolbox

Plan: create a small FastAPI execution server beside the robot and MoveIt infrastructure. Its public job is only to execute a verified pending plan after confirmation. It should not expose moveJ, moveL, servoJ, or generic pose-motion commands to the agent.
Public API

Start with GET /health and POST /execute. The execute request contains plan_name, confirmed_by, confirmation_text, and robot_name.

Backend v1

Execute by exact plan name through the existing Vizor route: publish std_msgs/String to /UR10/command/execute. This uses the trajectory already stored by MoveIt/Vizor.

Backend v2

If direct RTDE execution is required, add a trajectory extractor and convert the verified MoveIt plan into Robot MCP execute_trajectory(frames) behind the same narrow endpoint.

Pipecat agent changes

What must change before the agent can safely execute

Area Current state Required change
Robot context Stores recent executable plan names with timestamps. Become the source of policy truth for pending plans: source tool, source job, plan name, args or summary, turn id, user text, timestamp, and consumed state.
Job board Stores worker jobs, events, raw completion results, and failures. Remain the async ledger. Do not make Task Policy infer execute permission directly from job-board history.
Task Policy Requires fresh pose and recent executable plan for moveit_execute_plan. Also require explicit current-turn execution intent.
Completion bridge Can remember inline plan results, but queued planning jobs do not reliably feed pending-plan context. Observe successful worker completions, extract raw.plan_name, update RobotContextStore, and consume pending plans on accepted execute.
Execution adapter Robot MCP can execute RTDE trajectory frames, while MoveIt plan results expose named-plan metadata. If the real-robot tool uses Robot MCP, retrieve or retain the full planned trajectory and convert it to execute_trajectory(frames). If it uses Vizor, execute by exact plan name.
Physical readiness Fresh pose and recent plan checks exist; physical-mode and stop-state gates are not represented in the current policy finding. Before any RTDE-backed execute, check physical mode, robot-name/topic alignment, emergency stop, and protective stop status. Return a structured blocker if any signal is unavailable.
Prompt Allows plan-and-execute tools for simple actions. Revise instructions so planning and execution are separated unless the explicit-confirmation policy is intentionally relaxed.
Operational preflight

Make sure port 8766 is actually Robot MCP

Useful gotcha from the older page: the intended Robot MCP URL is http://127.0.0.1:8766/mcp/, but that port can be occupied by a plain Python static server. Before real-motion testing, verify the owning process and the MCP initialize response. 
Get-NetTCPConnection -LocalPort 8766 | Select-Object LocalAddress,LocalPort,State,OwningProcess
Get-Process -Id <pid> | Select-Object Id,ProcessName,Path
