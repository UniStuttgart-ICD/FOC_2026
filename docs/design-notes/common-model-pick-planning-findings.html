Planner comparison

Cartesian proves less than reachability. RRTConnect proves only reachability.

The current pick workflow uses Cartesian waypoints for approach, pre-grasp, close, attach, and lift metadata. RRTConnect was used only as a sampled planning probe for robot poses.

CartesianStrict straight waypoint paths. Good signal when complete, but fragile near assembled contacts.
RRTConnectJoint-space sampling can route to approach and pull-out poses that Cartesian rejects.
CaveatRRTConnect results here are plan-only. They do not close the gripper, attach the beam, or test object extraction.

Important limitation: the RRTConnect probes check robot pose reachability, not successful beam removal. A real pick/removal workflow must close the gripper, attach or otherwise model the target beam as moving with the gripper, and then test extraction.

Collision Reading

The planning scene treats each beam as a hard collision object. Intentional contact in the assembled frame still constrains MoveIt unless the workflow removes, attaches, or allows contact for the manipulated object.

Question Current reading Evidence
Are vertical elements failing because horizontals sit on top? Most likely, yes. All three vertical top Cartesian pick checks failed in the full structure. Earlier isolation checks succeeded after removing the horizontal beams.
Is that also why horizontal picks fail? Partly, depending on face and pull direction. Some horizontal grasps still succeed. Failures cluster around paths that pass through adjacent beam geometry or tighter gripper corridors.
Why can RRTConnect succeed where Cartesian fails? It can sample a curved joint-space route to the same pose. element_04/top, element_04/front, element_04/left, element_05/front, element_06/front, and element_06/left failed in Cartesian but planned in RRTConnect.

Recommended Next Test

The useful next workflow is hybrid: sampled free-space approach, explicit grasp state change, constrained extraction, then sampled retreat.

approach

Use OMPL/RRTConnect

Plan from the current state to a standoff pose so the arm can route around the assembled frame.

grasp

Close and attach

Close the gripper, then attach the target object or define allowed touch links before extraction.

extract

Constrain the pull

Use a short Cartesian pull along the selected removal direction, followed by a sampled retreat after clearance.

Implementation Plan

Add a semantic planner choice to moveit_plan_pick as planning_strategy. The agent may choose a strategy, but the MoveIt layer remains responsible for turning that strategy into real planner behavior.

contract

planning_strategy

Allowed values are auto, cartesian, and sampled_approach. Default to auto. Use cartesian for straight, constrained approach or lift behavior. Use sampled_approach when clutter blocks the straight path or the user asks for the random/sampling technique.

backend caveat

Do not fake sampling

Current pick planning calls the Cartesian planner. Current Vizor free motion is PTP-style target planning, not enough by itself to mean sampled pick approach. If sampled backend support is missing, return a structured unsupported-strategy failure instead of silently downgrading to Cartesian.

pipecat

Validate the argument

Update server/robot_control/call_validation.py so moveit_plan_pick accepts only the three strategy values. Add tests for valid, omitted, and invalid values.

prompt

Teach the agent when to choose

Update server/agent_control/prompt_parts/robot_contract.md: auto by default, cartesian for straight pulls, sampled_approach for cluttered or explicitly random-technique asks.

bridge

Forward without interpretation

Keep Pipecat validation local, then forward planning_strategy unchanged through the MCP bridge. The robot layer, not the prompt, resolves strategy into planner behavior.

moveit mcp

Add schema and routing

Add the parameter to moveit_mcp/server.py and moveit_mcp/tools.py. Resolve auto truthfully, preserve current behavior for cartesian, and report planning_strategy_resolved in results.

sampled path

Implement or block clearly

Add a real sampled approach backend path only if Vizor or MoveIt MCP can plan approach and pre-grasp with sampled semantics. Otherwise return status="planning strategy unavailable" with a retry correction.

verification

Prove the contract

Cover validation, prompt text, bridge forwarding, MCP strategy metadata, and sampled unsupported behavior. Live checks should call omitted, Cartesian, and sampled strategy variants.

Strategy Agent meaning First implementation behavior When to use
auto Let the robot layer choose the best available pick planner. May resolve to cartesian, but must say so in raw.planning_strategy_resolved. Default for normal pick requests.
cartesian Use the current waypoint-based pick workflow. Same behavior as current moveit_plan_pick. Straight pulls, vertical lifts, constrained extraction, and debugging baseline tests.
sampled_approach Use sampled planning to reach approach and pre-grasp poses around clutter. Either call a real sampled backend path or fail with a structured unsupported-strategy result. When Cartesian fails in clutter, or when the user asks for the random/sampling technique.
