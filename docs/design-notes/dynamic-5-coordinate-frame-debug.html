Short version

What actually happened

There were two bugs that looked like one bug. First, the element was being interpreted through the wrong coordinate assumptions, so the robot planned toward the opposite side of the real element. Second, the successful retry closed the gripper at a conservative height and never made RViz carry the element with the tool.

1. Source geometry

common_model.json says the model lives in rhino_world.

2. Planner frame

MoveIt plans in base_link. The scene objects must be published there.

3. Coordinate fix

The active mapping now swaps axes and flips signs with zero calibration offset.

4. Remaining gap

The gripper closed 6 cm above the top face; contact height still needs tuning.

Frame mismatch

Why correcting coordinates fixed the opposite-side motion

Coordinates are not universal numbers. x=0.8, y=0.26, z=0.28 only means "the top of dynamic_5" if both systems agree what x, y, z, and the origin mean. They did not. The Vizor/Rhino/Unity side uses one world; MoveIt uses the robot base. Without the transform, the planner receives plausible-looking numbers that point to the wrong physical side.

Bad path: plausible numbers, wrong frame Good path: coordinates converted into robot frame 
active mapping:
robot_x = -source_y
robot_y = -source_z
robot_z =  source_x

archived DF2025 offsets, disabled:
offset_x = -0.173
offset_y =  0.051
offset_z =  0.103

What those equations mean

A source-frame "forward" direction was not robot-frame forward. The mapping says: use source y to make robot x, use source z to make robot y, and use source x to make robot z. The minus signs are the mirror/rotation correction. The offsets were from an older calibration pass and are now kept only as archived reference values, not applied by default.

Where the swap likely comes from

This looks like a coordinate-convention bridge between the authoring/display world and ROS, not a measured translation error. In the observed correction, source x behaved like robot height, source y behaved like negative robot x, and source z behaved like negative robot y. That pattern is what you get when two systems disagree about which axis is up/forward and which direction is positive.

Base vs base_link

The frame name also mattered

The Vizor world pose was being stamped with base, while MoveIt reported its planning frame as base_link. In this Docker stack, those are not harmless aliases. They can differ by a rotation. Publishing world geometry in base_link removes that ambiguity.

Orientation research

Where the remaining mismatch can enter

The online docs and code scouts point to the same answer: the position bug was a source-frame to base_link conversion problem, while the orientation bug is a TCP-frame problem. MoveIt does not infer "grab this element"; it receives a pose for the configured end-effector link and treats the quaternion literally.

Confirmed fix Remaining orientation risk TCP / gripper frame
Evidence Finding What it means here
ROS REP-103 ROS standard frames are right-handed; body axes are x forward, y left, z up. The robot planning frame expects ROS-style axes, not authoring-tool axes.
Unity manual Unity uses a left-handed frame: x right, y up, z forward. Position and orientation need a basis conversion. A copied quaternion is suspicious.
Rhino guide Rhino separates fixed world coordinates from per-viewport construction planes. rhino_world should stay a named source frame until converted once.
UR frame docs base has the same origin as base_link, but is rotated 180 degrees about Z. The earlier base stamp can produce the exact opposite-side symptom.
MoveIt commander Pose targets are end-effector poses; 7-float targets are x, y, z, qx, qy, qz, qw. The quaternion controls the TCP frame. MoveIt does not know which way the jaws should face.
Local SRDF/URDF ur10_with_gripper.srdf uses base_link -> tcp; the patched TCP is 0.138 m past the Robotiq base. Height and orientation must be reasoned about at tcp, while attach currently uses tool0.

Likely source of the orientation bug

For manual Vizor targets, vizor_mcp/tools.py converts position but passes orientation through unchanged. That is not a valid Unity/Rhino to ROS quaternion conversion. For pick targets, the code is safer because it derives orientation from grasp context instead of copying the source quaternion.

Remaining pick-orientation risk

moveit_mcp/scene.py derives grasp faces from axis-aligned bounds. That gives top/front/side normals, but loses true object-local yaw. The gripper can approach from the correct side while its jaws are still rotated wrong for the beam.

Gripper gap

Why it closed above the element

This one is on the debug run, not the coordinate fix. After the first lower plan failed, the retry used a conservative low target: top_z + 0.06. Since dynamic_5 top is z=0.28, the gripper closed at z=0.34. That is exactly the "few centimeters above" you saw.

Why you did not see it hold the element

During the original run, attach_object only updated MCP simulated gripper state. It did not publish a MoveIt attached collision object, so RViz still saw dynamic_5 as free.

What changed after this note

moveit_attach_object now reads the free object from the planning scene and publishes a /planning_scene diff that attaches it to tool0 with gripper touch links. That is the RViz-visible path; the next live run should verify that the mesh follows the TCP.

Next fixes

What should change next

The coordinate correction tells us the main frame problem is solved. The code now keeps the old offsets disabled, derives pick orientation from the selected face, and publishes attached collision objects for RViz. The remaining work is contact-height proof plus one orientation refinement: preserve object-local jaw yaw.

  • Transform exactly once Keep rhino_world / Vizor inputs distinct from base_link robot targets. Current default uses axis swap/sign flip with no offset.
  • Face-aware pick orientation moveit_plan_pick now maps the selected grasp-face normal into a gripper orientation. The top face uses the working gripper-down quaternion.
  • Add object-local jaw yaw Current grasp faces come from axis-aligned bounds. Next, carry the element's local long/short axis into the grasp context so the jaws rotate with the beam.
  • Do not copy source quaternions Manual Vizor targets currently convert position only. If we use their orientation, convert it through the same source-to-ROS basis first.
  • Tune contact height Move from top_z + 0.06 to a measured contact offset. For top grasp, that is likely near top_z + 0.01 to top_z + 0.02, depending on the gripper TCP definition.
  • Visible attach moveit_attach_object now publishes a MoveIt attached collision object on /planning_scene. Live RViz proof should confirm it moves; if it appears offset, compare tool0 attach with tcp.
  • Separate planning from grasp proof A plan reaching the right x/y proves the frame fix. A grasp requires contact height, gripper closure, attachment, and post-lift object motion evidence.
  • Run HITL slow approach Use a human-watched slow descent for dynamic_5: stop at 6 cm, 3 cm, 2 cm, 1 cm above the top face, then choose the first safe visible contact height.
Debug checklist

How to read the next run

If the robot reaches the correct side, the coordinate frame is behaving. If it hovers, tune grasp height. If it lifts but the object stays behind, fix attach. Those are three different failures now, which is good news: they can be tested independently.

Question Pass condition If it fails
Is dynamic_5 in base_link? Object context reports planning_frame: base_link. Republish scene with the corrected frame path.
Does TCP reach the correct x/y? Final pose has x=0.800, y=0.260 near the element center. Recheck transform direction and offsets.
Does gripper close at contact? Close pose is near measured contact height, not top_z + 0.06. Tune TCP/gripper offset and approach height.
Are the jaws aligned with the beam? The TCP approach axis points into the selected face and the gripper jaw axis follows the element's local yaw. Carry object-local axes into grasp_faces; face normal alone is not enough.
Did a manual Vizor orientation get used? Any source quaternion is transformed into base_link before it becomes a MoveIt target. Ignore source orientation for pick, or implement a real quaternion basis conversion.
Does RViz show the element moving? The object becomes attached to tool0 and follows the TCP during lift. Inspect the /planning_scene diff and attached object link/touch links.
