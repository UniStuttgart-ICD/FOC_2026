Current State

The shipped fix is a narrow safety layer on top of the existing blackboard worker. It does not add the full task ledger yet.

What already existed

The agent queues long-running MoveIt work on the Robot Job Blackboard. The Robot Job Worker executes the exact queued call and reports completed or failed.

RobotJobBoard RobotJobWorker job_queued -> job_failed

What changed now

Internal recovery turns carry a flag that blocks automatic execution of a pending plan. A failed execute job reports the blocker directly.

allow_pending_plan_execution = false moveit_execute_plan failure -> blocker text planning failure -> guarded recovery

What comes next

The task ledger can add user-intent ownership above jobs: task ids, parent-child retries, cancellation, and stale-recovery suppression by task.

turn_id -> task_id -> job_id continuation vs conflict coordinator leases
Plain-language read: this builds on the blackboard. The worker still does deterministic robot work; the new gate controls how the agent reacts to worker failure facts.

Coordinator Decisions

The same failed background job is handled differently depending on what the user says next. That routing should be deterministic before the LLM is called.

Module Changes

  • Add allow_pending_plan_execution to AgentTurnInput.
  • Gate pending-plan auto-execution after robot observation.
  • Return deterministic text for failed moveit_execute_plan jobs.

What It Proves

  • Worker failure facts no longer become hidden execution approval.
  • The blackboard worker remains deterministic.
  • The task ledger can be added as a layer above this gate.

Does Not Solve Yet

  • No task_id ownership yet.
  • No real job cancellation or robot preemption.
  • No full concurrent command coordinator.

Code Shape

Keep the current job board deterministic. The shipped gate sits between worker events and graph recovery; the future task ledger sits above jobs.

AgentTurnInput

Carries whether this turn may auto-execute a pending plan. Normal user turns default to yes; internal recovery turns can say no.

user_text messages allow_pending_plan_execution

RobotJobBoard

Still owns exact MoveIt calls. The worker must not repair, reinterpret, or merge commands.

job_id tool_name arguments terminal event

LangGraph Gate

Checks the turn flag before executing the latest pending plan after observation.

observe_current_pose if allowed: execute_pending_plan else: call_model

Recovery Turn

Planning-job recovery can still ask the graph for help, but it disables pending-plan auto-execution.

source = failed_job allow_pending_plan_execution = false execute failure = blocker text

Event Grammar

Current worker events are enough for the safety gate. The task events are the future vocabulary for full concurrent command handling.

job_queuedCurrent: a MoveIt tool call entered the worker queue.
job_completedCurrent: the worker produced a terminal success event.
job_failedCurrent: the worker produced a terminal failure event.
recovery_guardedCurrent: internal recovery may not auto-execute a pending plan.
task_createdFuture: a user turn becomes tracked intent.
task_supersededFuture: a newer command replaces older recovery.
