Design note / execution robustness

Worker-Backed Plan Execution Hardening

The latest logs showed that planning succeeded, but execution did not reliably reach MoveIt. The fix makes execution use the same worker path as MoveIt planning, binds explicit execute requests to the latest pending plan, and makes worker-side candidate search execute the first successful plan instead of continuing to collect alternatives.

May 15, 2026 Pipecat agent MoveIt MCP Robot job worker

Observed Failure

Plans were generated, then execution was blocked as stale or unknown.

Around 13:20 local time, the trace showed a successful plan result with can_execute: true, followed by Task Policy rejecting moveit_execute_plan because no valid recent plan was known to context.

Root Cause

The completed worker plan could miss the next agent turn.

Plan results were recorded through the notification pump. If the user immediately said "execute it", robot context could still lack the returned plan_name.

Fix

Flush completed jobs before each turn, then let the worker execute first success.

Explicit execute language now routes to a deterministic graph node that queues moveit_execute_plan for the latest pending executable plan. For action requests, a successful worker plan now queues execution immediately.

Execution Flow

First Success Wins

For motion/action requests such as "move up" or "reach in my direction", the graph now stops after queueing one planning job. When the worker receives the first successful plan result with feedback.can_execute: true, it submits moveit_execute_plan at the front of the queue.

Candidate Cleanup

If more plan candidates were already queued for the same turn, the job board marks those queued planning jobs as cancelled. This keeps the worker from planning more after a valid candidate has already been selected for execution.

Evidence From The Run

  • Voice The user said "execute it" and then clarified "execute the plan that was completed just now".
  • Trace moveit_plan_free_motion returned a plan with can_execute: true.
  • Policy moveit_execute_plan was rejected as unknown or stale plan.
  • Docker The demo logs showed planning requests, but no matching UR10 executing line for that run.

Design Decision

Treat "execute it" as a stateful continuation of the latest executable plan, not as a fresh LLM tool-selection problem. The graph still observes pose first when context is stale, then checks for a recent pending plan and queues execution through the worker.

This keeps robot movement routed through MoveIt workflows and avoids adding local movement safety claims. The local change is orchestration reliability, not motion validation.

Remaining Boundaries

  • No plan-only run Planning-only requests such as "do not execute" still stop after planning.
  • Freshness Execution uses a recent pending plan only, then consumes it on success.
  • Safety MoveIt remains responsible for planning and execution validity.
  • Full checks Repo-wide Ruff and Pyright still report unrelated pre-existing issues outside this fix.

Net Effect

The agent no longer needs to rediscover or restate a plan name after planning completes. Completed plan results are made visible before the next turn, explicit confirmation sends a validated moveit_execute_plan job through the worker, and action-request planning now executes the first successful candidate while skipping remaining queued candidates.
