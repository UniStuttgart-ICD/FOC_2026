Robot Control Change Note

Geometry-Grounded Pick-Place

The agent now reasons over a paired world model: physical elements from MoveIt setup and user-positioned hologram targets from AR. It still observes and plans through MoveIt, but it receives the compact geometry context needed to choose the correct target pose.

What Changed

The change keeps geometry reasoning local, explicit, and small enough to refresh every turn.

Geometry World Context

GeometryWorldContextStore reads both shared geometry files on each render and injects paired dynamic identity plus hologram target poses into LangGraph instructions.

Canonical Dynamic Names

Boundary input like dynamic_05 normalizes to dynamic_5. Prompt examples and user-sensing candidates use unpadded dynamic_5-style names.

Strict Motion Semantics

The physical JSON is context, not live pose authority. MoveIt/RViz supplies source pose; the hologram JSON supplies desired object pose; the planner derives TCP and release poses.

Dynamic Roles

Verified release/place proof syncs physical pose bookkeeping. Structural/contact roles change only through geometry_update_dynamic_role when the role is clear or confirmed.

Runtime Flow

The model gets world context first, then uses robot tools for live grounding and execution.

World Model Paired geometry files

physical_model.json and hologram_model.json are paired by canonical element name.

Agent Context LangGraph instructions

Required Tool Path

Hologram-guided relocation stays a single compound pick-place workflow.

  1. ObserveCall moveit_list_scene_objects and confirm the canonical object exists.
  2. GroundCall moveit_get_object_context for dynamic_1 or the selected canonical dynamic object.
  3. PlanCall moveit_plan_compound_task with backend="mtc", requirements.goal="pick_place", requirements.object_name, and requirements.target_pose from Geometry World Context.
  4. ExecuteUse moveit_execute_task_plan only after explicit execution intent bound to the returned task solution.
  5. VerifyVerify release before claiming that the object moved or was placed; pose sync is deterministic bookkeeping after that proof.

Guardrails

The implementation intentionally avoids fallbacks and hidden geometry execution paths.

Hard Blockers

Missing files, unpaired bodies, duplicate canonical names, or invalid hologram poses render a blocked Geometry World Context. The agent must ask for an updated hologram target.

Not Added

No new MCP tool, no synthetic target-pose tool, no fallback to current object pose, and no hologram collision geometry in RViz/MoveIt.

Role Semantics

Roles are supporting_column, beam_supported_by, or unassigned. The agent must not infer role from pose alone; if uncertain, it asks the human.
