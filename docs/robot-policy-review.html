Current robot policies, in plain language

This page lists the rules that guide, block, or verify robot tool use. It is meant to help decide which policies are still needed and where they overlap.

Scope
Current working tree reviewed on 2026-05-18.
Focus: robot Task Policy, Robot Call Validation, execution approval, MoveIt/Vizor execution gates, geometry rules, and prompt rules.

Short Answer

What is solid

The main layering is clear: Task Policy checks simple preconditions, Robot Call Validation checks tool shape, and MoveIt or Verified Execution proves motion results.

What needs review

Release rules and some prompt-only instructions still need review. The task-planner fresh-pose question is settled: backend observation is enough.

Main risk

Some policies are duplicated, while a few important claims are only prompt guidance. That can make it hard to know which rule actually wins.

Runtime Flow

The strongest policies happen in this order. A later layer should not need to guess what an earlier layer meant.

1. Agent setup Shows only a small tool set to the model and injects Robot Context, Geometry World Context, and prompt rules.
2. Task Policy Blocks obvious bad order: no stale pose for low-level motion, no blind plan execution, no direct internal contract tools.
3. Call Validation Rejects unsupported tools, wrong robot names, wrong arguments, bad poses, bad targets, bad timeouts, and invalid task goal shapes.
4. MoveIt MCP Plans or executes through MoveIt. It records executable plans and task solutions with evidence.
5. Approval gates Task execution must use the exact recent task_solution_id and a matching approval payload.
6. Proof Execution success needs explicit verification, such as passed motion checks, attachment proof, or release proof.

Task Policy Details

Task Policy is small by design. It should answer: "Is this robot step obviously out of order before we even validate the tool shape?"

Fresh pose before low-level motion

For moveit_plan_free_motion, moveit_plan_cartesian_motion, moveit_plan_place, and moveit_execute_plan, the context must have a recent robot observation. The default window is 15 seconds.

  • Needed because relative or state-dependent motion should not use old robot state.
  • Decision: backend-side observation is enough for moveit_plan_manipulation_task. Task Policy does not need a second fresh-pose gate for that model-visible task planner.

No blind plan execution

moveit_execute_plan needs a recent executable plan name. The default plan window is 120 seconds. It also needs explicit execution language from the user.

  • Needed because the model must not invent plan names.
  • Needed because planning and execution are separate user-visible steps.

Manipulation task routing

If the user asks for pick, place, release, gripper-plus-motion, or another multi-stage manipulation, Task Policy redirects legacy free/cartesian/pick/place planners to moveit_plan_manipulation_task.

  • Needed because staged manipulation needs object, gripper, attach, release, and proof steps.
  • Review because this rule uses simple text terms such as pick, place, and then. It is limited, but it is still text-based routing.

Attach ordering

moveit_attach_object needs a non-empty object name and recent evidence that the gripper is closed. The default gripper-state window is 30 seconds.

  • Needed because attach is a planning-scene state change.
  • It should stay a simple ordering rule, not a proof that the object is physically safe to hold.

Contract-internal tools

moveit_release_object, moveit_verify_released_object, and moveit_remove_scene_object are reserved for cached execution-contract steps. Standalone model calls are blocked.

  • Needed so the model cannot detach or cleanup scene objects as a shortcut.
  • Duplicated in Robot Call Validation. This is probably harmless, but it is still duplicate policy.

Structured feedback

Task Policy returns ok=false, error, correction, retryable, and sometimes suggested_next_tool. The model sees this as tool feedback.

  • Needed because the next step should be explicit, not hidden in prose.
  • Good pattern to reuse for future policy blocks.

Hard Gates By Responsibility

Task Policy should own

  • Fresh observation before low-level motion.
  • Explicit user execution intent.
  • Recent returned plan or task solution evidence.
  • Simple step ordering such as close gripper before attach.

Robot Call Validation should own

  • Allowed tool names.
  • Allowed argument names.
  • Allowed robot name.
  • Finite poses and workspace bounds.
  • Supported goal names, target requirements, timeouts, and execution-contract schema.

MoveIt and Verified Execution should own

  • Whether motion is feasible.
  • Whether a plan has trajectory points.
  • Whether a command was published.
  • Whether final state matched expected state.
  • Physical actuation and state sync.

Prompt rules should own

  • How to ask clarifying questions.
  • How to describe results briefly.
  • How to prefer task-level tools.
  • How to choose clear defaults when hard validation does not decide.

Review Items

These are the places most worth reviewing before deleting or changing policies.

1. Release policy is not consistent

CONTEXT.md says release should require fresh robot state and held/attached evidence. Current Task Policy tests allow moveit_plan_manipulation_task release without prior robot state and without held-object evidence.

Decision needed: either make release evidence a hard gate, or update the docs to say the backend planner is responsible for checking release readiness.

2. Some important prompt rules are not hard gates

The prompt says to retry a correction once, ask when geometry is missing, avoid fallback targets, and not infer structural role from pose. Some of this is hard-coded, but some is only instruction text.

Decision needed: promote any must-not-break rule from prompt text into validation or backend checks.

3. Contract-internal tool blocking is duplicated

Task Policy and Robot Call Validation both block direct calls to release/verify/remove scene tools unless they are execution-contract steps.

Decision needed: keep both only if the earlier Task Policy error is useful. Otherwise, consolidate this in validation.

4. Task execution names are hard to compare

The system contains moveit_execute_task, moveit_execute_task_plan, moveit_execute_task_solution, and moveit_execute_plan. The model mainly sees moveit_execute_task, but internal paths still branch.

Decision needed: keep a short owner table for each execution name, or remove old compatibility paths once no caller needs them.

5. Job-worker auto execution bypasses the named Task Policy function

The worker can queue moveit_execute_plan after a successful planning job when the original user text clearly requested execution. It uses the returned plan and explicit text, but it does not call validate_task_step() for that queued execution.

Decision needed: either document this as a trusted worker shortcut, or call the same Task Policy gate from the worker.

6. Process Trace is correctly observational

Process Trace records policy blocks, validation, approvals, tool calls, and execution results. It does not appear to control behavior.

Decision needed: keep it out of runtime decisions.

Keep, Review, Or Remove

Recommendation Policies Reason
Keep Allowed tools, robot name, argument shape, pose bounds, explicit execution, recent plan names, task-solution approval, execution-contract proof, MoveIt/Verified Execution checks. These prevent unsupported calls or require concrete evidence before execution and success claims.
Review Release readiness, manipulation-task fresh pose, stale approval after newer user intent, job-worker auto execution, geometry target fallback rules. These have real behavioral meaning, but the current enforcement is split or partly soft.
Possibly consolidate Duplicate contract-internal tool blocks and old hidden execution/planning tool variants. Duplicate gates are not wrong, but they make policy ownership harder to review.
Do not treat as safety boundary Robot Call Validation, Task Policy, Prompt Contract, Process Trace, Geometry Context. The architecture says movement safety belongs to MoveIt planning/execution and the robot simulation or verified execution stack.

Plain Examples

"Move up a bit"

The graph observes the current pose. The model should plan a free or cartesian move. Task Policy requires that fresh pose. Call Validation checks the pose shape and bounds. MoveIt decides if the motion is feasible.

"Execute the plan"

Task Policy checks that there is a recent returned plan_name and that the user explicitly asked to execute. MoveIt or Verified Execution then proves whether execution actually passed.

"Pick up beam 2"

The prompt and visibility rules push this to moveit_plan_manipulation_task. Validation checks the goal and object name. The planner returns a task_solution_id and approval payload. Execution still needs explicit approval.

"Let go"

This is the clearest review case. Docs say release needs fresh state and held-object proof. Current Task Policy allows the model-visible release plan without that prior proof. Decide where release readiness should be enforced.
