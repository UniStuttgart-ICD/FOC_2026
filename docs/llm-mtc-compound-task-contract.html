LLM and MTC Compound Task Contract

The LLM listens to the user and describes the desired manipulation task. MTC owns the robot task construction and solving. The real robot executor only runs a solved, approved, proof-backed contract.

Plain rule

The LLM gives intent and constraints. MTC decides whether the robot can do it. Execution starts only after MTC returns a solved task solution.

LLM agent

Understands the request

It turns "move this there and let go" into a structured goal with object, target, and preferences. It may suggest hints, but those hints are not executable robot instructions.

MCP / MTC

Builds and solves the task

It validates the goal, grounds the object in the scene, creates the MTC task, searches for a solution, and returns clear blockers when the task cannot be solved.

Executor

Runs only solved contracts

It executes the ordered contract from the backend: motion, gripper close/open, attach/release, and verification proof. It does not invent missing stages.

Data Flow

Planner Input Shape

The clearest public contract is requirements plus preferences. Requirements are hard facts needed for the task. Preferences are hints the planner may use or reject.

{
  "requirements": {
    "goal": "move_and_release",
    "object_name": "beam_1",
    "target_position": {
      "x": 0.5,
      "y": 0.2,
      "z": 0.3
    }
  },
  "preferences": {
    "grasp_face": "top",
    "retreat_after_release": true
  }
}

Why this is safer

  • The LLM does not author executable robot stages.
  • MTC can derive the real stage graph from the goal.
  • Planner failure gives useful feedback to the LLM.
  • Unsupported work such as slide or push can fail early.

How the Retry Loop Works

1

LLM submits task requirements

Example: hold beam_1, move it to the target, then release it. Optional preferences can describe grasp face, retreat, or orientation.

2

MTC accepts or rejects

If it cannot solve the task, it returns a failed stage, blocker, and correction. It does not return a task_solution_id for unsolved work.

3

LLM adjusts only the request

The LLM may change preferences or ask for a different target. It still cannot bypass MTC or send raw waypoints to the real robot.

What MTC Must Return Before Execution

task_solution_idExact solved task to approve and execute.
execution_contractTyped ordered steps for the executor.
stage evidencePlanner stages, costs, candidates, failed blockers.
scene snapshotObject and world state used by the solved plan.

Execution Boundary

  • moveit_plan_compound_task should call MTC and return either a solved task solution or a structured failure.
  • moveit_execute_task_plan should execute only a recent approved task_solution_id with a backend-issued execution_contract.
  • Gripper close/open must be verified before attach or release state changes are trusted.
  • After release, the system must verify the object is released before saying the object was placed, dropped, or let go.
  • No automatic fallback to emulated, legacy, waypoint, slide, or push execution belongs in this path.

Terminology

Requirements

Hard task facts: goal, object, target pose or position, and any must-have constraints.

Preferences

Planner hints: preferred grasp face, retreat behavior, orientation style, or other choices MTC may reject.

Stage intents

Optional semantic hints such as approach, hold, move, release. They are not executable stages and should not be required for every call.

Execution contract

The backend-issued proof contract the real robot executor follows after explicit approval.
