Pipecat robot control research

Rank the robot layer, not the model.

The strongest pattern across the local design notes, ARCHITECTURE references, MoveIt docs, and three parallel research lanes is simple: keep the agent in charge of sequencing, but make every robot step stage-aware, schema-shaped, approval-gated, and verified against the planning scene.

Executive Summary

The current architecture is directionally right. The useful improvements are not broader ROS access or more model freedom. They are tighter robot-side contracts that make the next valid action obvious after each observation, plan, failure, execution, or verification event.

Keep

Agent sequencing

The agent should still choose the next semantic tool: observe, ground, plan, execute only a valid plan, verify, or ask for clarification.

Deepen

Robot-owned evidence

MoveIt planning results should report stage names, failed stage, candidate counts, scene snapshot, solver, cost, clearance, and next admissible tool.

Reject

Hidden motion shortcuts

No broad ROS mutation tools, no combined plan-and-execute verbs, and no deterministic text-to-motion fallback outside the model/tool loop.

Most useful bundle: add typed result schemas and stage-aware failure fields first, then add scene snapshot gating and object-state proof, then add deterministic execution approval and replay artifacts.

2026-05-15 addendum: the concrete first fix is to carry raw.next_action.after_success through queued jobs. That turns the prior staged-workflow recommendation into an executable compound chain: moveit_plan_pick(auto) can execute its preposition plan, enqueue the local pick continuation, run staged grasp/attach/lift, and prove object state before success. See compound workflow chaining research and implementation plan.

Current workflow pressure

Cartesian reachability is not object manipulation.

The common-model findings showed that sampled planning can reach more approach and pull-out poses than Cartesian planning, but it still does not prove that a held object cleared the structure. Object state must be verified separately.

Local evidenceCartesian succeeded on 5 / 30 common-model pick faces; RRTConnect found 18 / 30 plan-only candidates.
InterpretationSampled planning helps the robot reach poses, but attached-object extraction is a separate proof.
Workflow rulePlan, execute, and verify motion; also verify gripper state and planning-scene object state.

Recommended Workflow Shape

This is the workflow the findings point toward. It does not require exposing a new broad workflow tool to the model. It can be implemented as stricter contracts around the existing semantic tools, plus queued after_success continuations for compound plans.

1. Observe

Fresh robot and scene state

Require moveit_get_current_pose, object list, and object context before motion, relative commands, retries, and pick/place.

2. Plan

Stage-aware candidate

Return plan name, solver, scene snapshot, stage report, candidate attempts, cost, clearance, and executable gate fields.

3. Approve

Exact plan confirmation

Bind execution to one recent plan_name, one user approval turn, and unchanged scene and robot readiness state.

4. Execute

Deterministic worker

The Robot Job Worker executes the exact validated tool call, preserves raw.next_action.after_success, and enqueues the continuation only after success.

5. Verify

Motion plus object proof

Verify final pose or joints, gripper state, attached object state, and eventually placed object state before claiming success.

6. Recover

One grounded next move

Failures return a specific failed stage, correction, retryability, and next tool. Recovery does not auto-execute.

7. Record

Replayable evidence

Save policy, validation, plan, approval, execution, verification, and terminal job event data in one turn artifact.

8. Evaluate

Scenario-level scoring

Model evals should score behavior through the whole embodied loop, not just final assistant text.

Suggested Build Sequence

This tab turns the ranked findings into five small plans. Each plan explains how the new contract expands the MoveIt tools already in the robot contract, without adding hidden execution shortcuts.

1. Typed manipulation plan contracts

Give every planner the same kind of result envelope, starting with moveit_plan_pick and moveit_plan_place. A plan should say what it moves, what state it depends on, whether it can be executed, and which exact tool should be called next.

Concrete benefit: for "place the red beam on the table," moveit_plan_place returns the attached object, target support, release TCP pose, stage report, raw.plan_name, and feedback.can_execute. The agent no longer invents a release pose or treats a Cartesian move as proof that the object was placed.
Open implementation plan

How this expands current tools

Keep moveit_plan_free_motion, moveit_plan_cartesian_motion, moveit_plan_pick, and moveit_plan_place as planning-only tools, but make their outputs share fields such as ok, plan_type, object_name, scene_snapshot_id, stage_report, failed_stage, correction, and suggested_next_tool.

Why first

Typed plan results are the base layer for every later gate. Without a common shape, approval, scene freshness, recovery, and replay all have to guess what a planner meant.

Implementation plan

  1. Define one shared plan-result schema in Robot Control.
  2. Apply it to pick and place first, then free and Cartesian motion.
  3. Require moveit_plan_place to include held object, target support or pose, orientation mode, and release evidence.
  4. Teach Robot Context to store typed pending plans by raw.plan_name.
  5. Describe future moveit_plan_hold as another planner using this same result shape.

Verification

  • Planning tools return typed results without moving the robot.
  • moveit_execute_plan rejects unknown, stale, or non-executable plan names.
  • Place tests require grounded object context and verified attachment before planning.
  • Failure cases include a failed stage, correction, and next suggested tool.

2. Scene snapshot and stage evidence

Stamp planning with the robot and scene state it actually used. The agent should be able to say, in plain terms, "this plan was based on this pose, these scene objects, and this selected object context."

Concrete benefit: if a human moves the red block after a pick plan was made, the execution gate sees that the scene snapshot changed and blocks moveit_execute_plan before the robot acts on stale geometry.
Open implementation plan

How this expands current tools

Wrap existing observation tools into one advisory snapshot: moveit_get_current_pose, moveit_get_robot_state, moveit_list_scene_objects, and moveit_get_object_context. Planning tools then echo the snapshot ID, pose age, selected object, solver, and stage evidence.

Why this helps

Failures become explainable. Instead of "planning failed," the system can say the side-face approach failed because clearance was blocked in the snapshot used for planning.

Implementation plan

  1. Create a compact scene snapshot record in Robot Context.
  2. Require a fresh snapshot before object-relative, pick, place, and hold planning.
  3. Add snapshot and stage fields to planning results and failure explanations.
  4. Reject execution when the pending plan snapshot no longer matches current scene state.

Verification

  • Object tasks call moveit_list_scene_objects before moveit_get_object_context.
  • Pick/place planning is blocked when object context is missing or stale.
  • Simulated tests reject stale-pose and stale-scene execution.
  • moveit_explain_motion_failure receives the snapshot used by the failed plan.

3. Explicit execution approval payload

Planning should never imply execution. Before moveit_execute_plan, the system should hold a small approval record that names the exact plan, object, expected movement, scene snapshot, and user approval turn.

Concrete benefit: after "move up five centimeters," the agent can say the pending plan is cartesian_up_05m_01. If the user then says "yes, do it," only that plan can be executed. If the agent tries cartesian_forward_01, Task Policy blocks it.
Open implementation plan

How this expands current tools

moveit_execute_plan already executes only a returned plan name. The approval payload adds intent binding: approved plan name, source planning tool, target object when present, expected displacement, snapshot ID, and approval timestamp.

Why this helps

Voice turns are easy to interrupt or redirect. Binding approval to one pending plan prevents a later recovery or clarification turn from accidentally executing an older plan.

Implementation plan

  1. Define the pending execution approval shape in Agent Orchestration or Robot Context.
  2. After any executable moveit_plan_* result, store one pending approval candidate.
  3. Allow moveit_execute_plan only when the approved payload matches the plan name and scene snapshot.
  4. Return structured correction text for missing approval, wrong plan, stale approval, or changed scene.

Verification

  • Execution without approval is blocked.
  • Execution with a different plan name is blocked.
  • Stale approvals are blocked after new user intent or scene changes.
  • Approved execution still routes only through moveit_execute_plan.

4. Object-state proof and moveit_plan_hold

Pick, place, and hold should be object-state workflows, not just end-effector motion. The agent should know whether the object is grounded, attached, released, or being held before it claims success.

Concrete benefit: for "hold this beam while I fasten it," the agent verifies the beam is attached, calls future moveit_plan_hold for a stable hold pose and duration, asks for approval, executes the returned plan, then verifies the object is still attached. It does not treat "the TCP moved there" as proof that the beam is secure.
Open implementation plan

How this expands current tools

Keep moveit_verify_attached_object after pick and add a future moveit_verify_placed_object after place. Add moveit_plan_hold as a planning-only tool for a verified attached object: it plans a stable pose or constraint, then returns a normal executable plan.

Pick execution should be staged: pre-grasp motion, gripper close, planning-scene attach, lift, and object-state verification. This is the object proof that compound workflow chaining makes reachable.

Why this helps

The workshop scenarios include human-robot construction work, where holding a part steady can be as useful as moving it somewhere. A hold plan gives the agent a clear tool for support tasks without overloading place or Cartesian motion.

Implementation plan

  1. Track grounded object, planned object, attached object, placed object, and verification timestamp in Robot Context.
  2. Block place and hold planning unless attachment is verified for the named object.
  3. Design moveit_plan_hold inputs: object name, hold pose or constraint, duration hint, allowed orientation drift, and support context.
  4. Keep hold planning separate from gripper commands and execution.
  5. Add placed-object verification before the agent claims the object was released at the target.

MTC boundary

Use MTC-inspired stages internally behind semantic tools. Do not add a raw arbitrary MTC tool first; a later backend="mtc" can sit behind moveit_plan_pick, moveit_plan_place, and moveit_plan_hold.

Verification

  • Pick cannot claim success until attachment verification passes.
  • Place cannot plan without the correct attached object.
  • Hold cannot plan for an unknown or unverified object.
  • Failed attachment or release returns correction text and a suggested next tool.

5. Eval and replay scenarios

Turn the workflow rules into repeatable checks. Eval and replay should prove that the agent observes, grounds, plans, asks for approval, executes only a returned plan, verifies object state, and recovers clearly when blocked.

Concrete benefit: a replay for "pick the red block" expects moveit_list_scene_objects, moveit_get_object_context, moveit_get_current_pose, moveit_plan_pick, approval, moveit_execute_plan, then moveit_verify_attached_object. If planning fails, the expected path is moveit_explain_motion_failure, not blind execution.
Open implementation plan

How this expands current tools

This adds a scenario runner around the current semantic tools. It does not add robot behaviors. It records expected tool order, typed tool outputs, policy decisions, validation results, approvals, and final assistant text.

Why this helps

Good robot-agent behavior is a loop, not one final answer. Replay makes regressions visible when a model skips grounding, executes too early, or claims success without verification.

Implementation plan

  1. Define compact scenario files with utterance, mocked robot state, mocked scene objects, expected tool sequence, and expected result.
  2. Extend the simulated MoveIt adapter to cover pick, place, hold, stale pose, unsupported sampling, and verification failures.
  3. Capture Process Trace and Live Eval Evidence fields in one readable replay artifact.
  4. Keep live MCP/RViz proof opt-in; deterministic replay should run in ordinary tests.

Verification

  • No scenario executes without a prior valid plan name and approval.
  • Object-relative tasks ground objects before planning.
  • Plan failures route to moveit_explain_motion_failure.
  • Replay output produces a readable diff when the sequence is wrong.
