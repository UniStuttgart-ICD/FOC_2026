Agent 2 research addendum

Compound tools need staged execution, not one more trajectory.

The last RViz run proved that reaching the object and manipulating the object are different contracts. The robot can execute free motion, and it can render a local Cartesian pick plan, but the current chain can still skip the gripper close, object attach, and verified lift boundary.

Observed plan attempts and a later execution of the returned pick plan.

Root gaps 2

Queued continuations are dropped; pick workflow steps are metadata only.

Recommendation MTC-shaped

Use MTC's stage model as the contract before requiring MTC as a dependency.

What Actually Failed

The failure was not that RRTConnect was missing from the Docker image. The benchmark already showed sampled planning is available. The failure was wiring: the agent and MCP server do not yet preserve the full compound workflow across execution boundaries.

Pipecat queue

Continuation metadata is not carried.

raw.next_action.after_success exists in the preposition plan result, and Robot Context can remember it. The queued worker currently keeps only robot_name and plan_name when it auto-queues moveit_execute_plan.

MoveIt MCP

Pick steps are descriptive.

build_pick_workflow() already lists close_gripper, attach_object, and lift. execute_plan() only has special object-state handling for workflow_kind == "place", not for pick.

Planning model

One Cartesian path cannot grasp mid-path.

The current pick Cartesian waypoints are approach, pre-grasp, lift. If that is executed as one trajectory, there is no safe boundary where the gripper can close and the object can be attached before lift.

Agent 2: MTC Pattern

MoveIt Task Constructor is the best reference model because it treats pick/place as staged manipulation, not as one opaque pose goal.

ObserveCurrent state plus planning-scene object context.
ConnectFree-space move to a pre-grasp or staging state.
MoveRelativeCartesian approach to a grasp boundary.
ModifySceneClose gripper, allow touch, attach object.
MoveRelativeLift or retreat with the object attached, then verify.

Decision: do not expose raw MTC stages to the agent first. Keep agent-facing tools semantic: moveit_plan_pick, moveit_plan_place, future moveit_plan_hold, and moveit_execute_plan. Internally, make those tools return and execute MTC-shaped stage traces.

Fix Shape

The existing research page is directionally right. The queue continuation fix is the concrete missing bridge between its approval/verification sequence and the live worker that actually runs robot jobs.

Fix Where Why it is needed
Carry after_success through queued jobs. server/robot_control/job_board.py and server/robot_control/job_worker.py Auto pick preposition can execute and then enqueue the local Cartesian pick instead of stopping after free motion.
Queue pick/place planning jobs when the user asks for action. server/agent_control/robot_job_submission.py and LangGraph tests Long robot action chains should continue in the worker beyond the model's tool-turn limit.
Split pick execution into stage boundaries. moveit_mcp/pick.py and moveit_mcp/tools.py The gripper must close and the object must attach between approach and lift.
Promote object-state proof. MoveIt MCP execution, Pipecat Robot Context, and eval scenarios Pick, place, and hold should succeed only after attachment, release, or hold state is verified.
Add optional MTC backend later. MoveIt MCP backend boundary MTC should be an implementation backend behind the same semantic contract, not a raw agent tool requirement.
