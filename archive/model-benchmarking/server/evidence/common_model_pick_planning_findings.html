Planner Modes

Current pick workflow

  • moveit_plan_pick builds approach, pre-grasp, close, attach, and lift steps.
  • The motion portion goes through the Cartesian planning path used by the Vizor/MoveIt stack.
  • This is strict: straight waypoint paths fail when the tool path is blocked by nearby collision objects.

Sampled comparison

  • OMPL is the default planning pipeline in the live stack.
  • The random/sampling probe used RRTConnect.
  • The probe was plan-only: current pose to approach pose, then approach-final-state to a pull-out pose.

Important limitation: the RRTConnect probe does not prove that the beam itself was removed. It checks robot reachability to approach and pull-out poses. A real removal workflow still needs gripper close, object attachment or allowed-touch semantics, and a controlled extraction segment.

Collision Interpretation

Question Answer Evidence
Are the vertical elements failing because horizontals sit on top? Yes, that is the most likely explanation. The vertical top faces are occupied by lower horizontal beams. In the full structure, all three vertical top Cartesian pick checks failed. Earlier isolation checks succeeded after removing the horizontal beams. RRTConnect can still reach some robot-only top poses, but that does not prove the attached vertical beam can move through the contact geometry.
Is that also why horizontal picks can fail? Partly, depending on face and extraction direction. Some horizontal grasps still succeed in the full structure, so the full structure does not make all horizontal picks impossible. Failures cluster around faces where the gripper path or intended lift/pull would pass through adjacent collision geometry.
Why can RRTConnect succeed where Cartesian fails? RRTConnect can sample a curved joint-space route to the same pose. For example, element_04/top, element_04/front, element_04/left, element_05/front, element_06/front, and element_06/left failed in Cartesian but planned in RRTConnect.

The planning scene currently treats each beam as a hard collision object. If two beams are intentionally touching in the assembled structure, MoveIt still sees those contacts as collision constraints unless the workflow explicitly removes, attaches, or marks allowed contacts for the object being manipulated.

Recommended Next Test

The best next workflow is hybrid rather than purely Cartesian or purely random:

Step Planner/action Reason
1. Approach OMPL/RRTConnect free-space plan to a standoff pose Lets the arm route around the assembled structure instead of requiring a straight line.
2. Grasp Close gripper, then attach the target object or allow touch links Prevents the target beam from being treated like an obstacle to the tool that is holding it.
3. Extract Short constrained Cartesian pull along the selected removal direction Keeps the physical pull-out meaningful and avoids random motion through contact.
4. Retreat OMPL/RRTConnect free-space retreat Allows a comfortable path away from the frame after the beam is clear.
